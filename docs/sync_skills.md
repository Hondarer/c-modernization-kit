# skill 同期とドキュメント生成連携

## 概要

この文書は、`make skills` の導入と、それを GitHub Actions / Jenkins のドキュメント生成へ組み込んだ変更内容を説明します。

今回の変更の主目的は、サブモジュール配下にある skill をプロジェクトルートへ集約し、Codex、Claude など複数の AI エージェントから同じ skill を確実に呼び出せるようにすることです。  

対象となる主なファイル:

- `makefile`
- `bin/sync_skills.sh`
- `.agents/skills/.gitignore`
- `.github/workflows/ci.yml`
- `.jenkins/inner-build.sh`
- `.jenkins/README.md`
- `docs/skill-guide/07-ci-cd/jenkins.md`

## 背景

skill の実体はルート配下だけでなく、サブモジュール側にも存在します。たとえば `framework/testfw/.agents/skills/create-testfw-mock/` は testfw 固有の skill 文書の正本です。

このままでは、AI エージェントが参照する skill 配置がルート側とサブモジュール側で分散し、また、あるエージェントでは見えるが別のエージェントでは見えない、という状態が起こりえます。  
そのため、まず skill をエージェント共通の参照位置へ同期する仕組みを自動化します。

その結果として、`make docs` でも `.agents/skills` を入力に使えるようになり、skill ドキュメントの統合発行も可能になります。

## 設計方針

### 正本の扱い

- ルート固有 skill はルート `.agents/skills` を正本とする
- サブモジュール固有 skill は各サブモジュールの `.agents/skills` を正本とする
- 同名 skill が衝突した場合は、ルート側とサブモジュール側の中身を混在させず、サブモジュール側で置き換える

### 配置先

- Codex / GitHub Copilot 向けの集約先: `.agents/skills`
- Claude Code 向けの参照先: `.claude/skills`

この 2 つを同期させることで、エージェントごとの skill 探索位置の違いを吸収します。

### プラットフォーム差分

- Linux: symlink を優先して重複コピーを避ける
- Windows: 実体コピーで配置する

この方針により、Linux では同期コストを抑えつつ、Windows ではリンク権限や互換性の問題を避けられます。

## `make skills` の実装

### 入口

ルート `makefile` に `skills` ターゲットを設けています。

```makefile
.PHONY: skills
skills :
	"$(BASH)" "$(CURDIR)/bin/sync_skills.sh"
```

### 実処理

実際の同期ロジックは `bin/sync_skills.sh` に集約しています。処理手順は次のとおりです。

1. `.gitmodules` からサブモジュール一覧を取得する
2. 各サブモジュールについて `.agents/skills` の有無を確認する
3. 前回生成したサブモジュール由来 skill を `.agents/skills/.gitignore` をもとに削除する
4. 現在のサブモジュール由来 skill を収集する
5. 同名 skill が複数サブモジュールに存在する場合はエラーにする
6. ルート `.agents/skills/<skill>` を再作成する
7. `.claude/skills` を作り直す

### Linux での動作

- `.agents/skills/<skill>` はサブモジュール実体への symlink として作成する
- `.claude/skills` は `../.agents/skills` への symlink として作成する

例:

```text
.agents/skills/create-testfw-mock -> ../../framework/testfw/.agents/skills/create-testfw-mock
.claude/skills -> ../.agents/skills
```

### Windows での動作

- `.agents/skills/<skill>` は実体ディレクトリとして再帰コピーする
- `.claude/skills` も `.agents/skills` から全量コピーする

### `.agents/skills/.gitignore`

サブモジュール由来 skill を Git 管理対象外にするために、`make skills` は `.agents/skills/.gitignore` を自動生成します。

例:

```gitignore
# Generated by bin/sync_skills.sh
/create-testfw-mock/
```

このファイルには 2 つの役割があります。

- Git に対して「この skill は生成物なので追跡しない」と伝える
- 次回同期時に「前回生成したサブモジュール由来 skill はどれか」を特定する

## GitHub Actions への反映

GitHub Actions では `publish-docs` ジョブのドキュメント生成前に `make skills` が呼び出されます。

順序:

1. サブモジュール初期化
2. `make skills`
3. `make doxy`
4. `make docs`

該当箇所は `.github/workflows/ci.yml` です。

```yaml
- name: Sync skills
  run: make skills

- name: Generate and publish documentation
  run: make doxy && make docs
```

これにより、CI 上でも `.agents/skills` と `.claude/skills` が最新状態になってから Markdown 発行が走ります。  

## Jenkins への反映

Jenkins 側はドキュメントそのものでは自動化されず、`.jenkins/inner-build.sh` が実処理を担います。  
そのため、`BUILD_DOCS=1` 時の流れは以下の通りです。

```bash
make skills 2>&1 | tee "logs/linux-${OS_NAME}-skills.log"
make doxy 2>&1 | tee "logs/linux-${OS_NAME}-doxy.log"
make docs 2>&1 | tee "logs/linux-${OS_NAME}-docs.log"
```

## 制約と注意点

1. `make skills` は `.gitmodules` を参照して対象サブモジュールを決めるため、サブモジュール定義の更新時はこの前提も意識する必要があります。
2. 同名 skill が複数サブモジュールに存在する場合、スクリプトは安全側で停止します。
