# ユニキャスト通信における dst_addr / src_addr / セッション識別仕様補足

## 概要

1:1 通信 (ユニキャスト) のサービス定義に `dst_addr` および `src_addr` フィールドを追加する。
各フィールドは送信者と受信者で以下のように解釈される。

| フィールド | 送信者 (potrSend 発行側) | 受信者 (potrOpenService でコールバック登録側) |
| ---------- | ------------------------ | --------------------------------------------- |
| `dst_addr` | パケットの送信先アドレス | bind するアドレス (購読アドレス) |
| `src_addr` | bind するアドレス (送信元アドレス固定) | 受け付ける送信元 IP フィルタ |

---

## ネットワーク層

### IPv4 専用

porter は IPv4 のみをサポートする。IPv6 は対象外とする。

### アドレス指定形式

`dst_addr` / `src_addr` には以下のいずれかを指定できる。

- IPv4 アドレス (例: `192.168.1.10`)
- DNS で解決できるホスト名 (例: `receiver.local`)

---

## DNS 解決ポリシー

### 解決タイミング

`potrOpenService()` 呼び出し時に 1 回だけ解決する。
解決結果はプロセス内にキャッシュされ、以後の送受信で再利用する。

### DNS 更新時の挙動

プロセス起動後に DNS レコードが更新されても、キャッシュされた解決結果は更新されない。
DNS 更新後に接続できなくなった場合は、プロセスを再起動して再解決すること。

**前提**: porter は静的ルート環境を想定する。動的な DNS 切り替えは仕様外とする。

### 複数アドレスが返された場合

DNS が複数のアドレスを返した場合の動作は仕様上未定義とする。
実装上は `getaddrinfo()` が返すリストの先頭 (インデックス 0) を採用する。

---

## 受信者の bind 挙動

受信者は `potrOpenService()` の内部で `dst_addr` を解決し、得られた IPv4 アドレスと `dst_port` の組み合わせでソケットを `bind()` する。

### bind 失敗時の挙動

以下のいずれかの場合、`potrOpenService()` は `POTR_ERROR` を返して終了する。

- `dst_addr` の名前解決に失敗した場合
- 解決されたアドレスが当該ホストの NIC に割り当てられていない場合
- `bind()` がその他の理由で失敗した場合

**注意**: `dst_addr` を受信者のホストに割り当てられているアドレスと一致させるのは、設定ファイルの利用者の責任とする。

---

## 送信者のアドレス・ポート挙動

送信者は `potrOpenService()` の内部で `src_addr` を解決し、そのアドレスでソケットを `bind()` する。
これにより送信元 IP アドレスが固定され、OS の経路選択に左右されない。

送信元ポートは OS に自動選定させる。`src_addr` に対して特定ポートでの `bind()` は行わない。

受信者側では `dst_port` で `bind()` したソケットを使用して ACK / NACK を返送するため、
再送制御上の返信元ポートは `dst_port` となる。これはプロトコル上自然な動作であり、
複数の送信者を `dst_port` 単位で識別しつつ複数サービスを運用することを可能にする。

---

## 受信者の送信元フィルタリング

受信スレッドは、受信したパケットの送信元 IP アドレスをサービス定義の `src_addr` と照合する。
一致しないパケットは破棄する。フィルタはアプリケーション層で実施し、ソケットの bind アドレスは変更しない。

### 1:1 通信の制約

1:1 通信サービスにおいて `potrSend` を発行するプロセスは 1 つでなければならない。
同一 IP アドレス上の複数プロセスが同じサービスの送信者となることは現行仕様では対象外とする。

**将来対応予定**: 同一 IP の複数送信者は `session_id` によって識別する拡張を予定する。
受信スレッドの IP フィルタ層を `session_id` による振り分けに置き換えることで対応する。

---

## セッション識別

### 目的

送信者プロセスの再起動後に OS が同じ ephemeral ポートを再利用した場合や、
ネットワーク上に残留した前セッションのパケットが遅延して届いた場合に、
受信者がそれらを誤って処理しないようにする。

### セッション識別子の構成

`potrOpenService()` 呼び出し時に以下の 2 値を決定し、全パケットに付与する。

| フィールド | 型 (ワイヤーフォーマット) | 決定方法 |
| ---------- | ------------------------- | -------- |
| `session_id` | `uint32_t` (NBO) | 乱数 |
| `session_start_time` | `struct timespec` 相当 (NBO、下記参照) | 現在時刻 |

#### session_start_time のワイヤーフォーマット

`struct timespec` をネットワークバイトオーダーで固定幅にエンコードした形式とする。

```c
/* ワイヤー上の session_start_time レイアウト (計 12 バイト) */
int64_t tv_sec;   /* POSIX エポックからの秒数 (NBO) */
int32_t tv_nsec;  /* ナノ秒部 0〜999,999,999 (NBO) */
```

### 受信者のセッション採用ポリシー

受信者は受信したパケットの `(session_start_time, session_id)` を
現在追跡中のセッションと比較し、以下のルールで採用・破棄を決定する。

```
1. session_start_time が現在追跡中より大きい  → 新セッションとして採用
2. session_start_time が等しく session_id が大きい → 新セッションとして採用 (タイブレーク)
3. それ以外                                  → 旧セッションのパケットとして破棄
```

採用判定はタイムスタンプを同一送信者の 2 つのパケット間で比較するため、
送信者と受信者のクロック同期には依存しない。

### service_id の照合

受信スレッドは受信したパケットの `service_id` をサービス定義の `service_id` と照合する。
一致しないパケットは破棄する。

これにより、同一の `dst_addr:dst_port` に誤って届いた他サービス向けパケットを排除できる。

### クロック逆行時の挙動

OS クロックが手動設定や NTP スラムで過去に戻った場合、
新セッションのタイムスタンプが旧セッションより小さくなりえる。
この場合の動作は仕様上未定義とする（静的環境前提の既知制限）。

---

## パケット構造への影響

`PotrPacket` に `service_id`・`session_id`・`session_start_time` フィールドを追加する。
これはプロトコルレベルの変更であり、既存の porter と互換性はない。

```c
typedef struct
{
    int32_t  service_id;        /* サービス識別子 (NBO)。受信時に照合する。 */
    uint32_t session_id;        /* セッション識別子 (NBO) */
    int64_t  session_tv_sec;    /* セッション開始時刻 秒部 (NBO) */
    int32_t  session_tv_nsec;   /* セッション開始時刻 ナノ秒部 (NBO) */
    uint32_t seq_num;           /* 通番 (NBO) */
    uint32_t ack_num;           /* 確認応答/再送要求番号 (NBO) */
    uint16_t flags;             /* パケット種別フラグ (NBO) */
    uint16_t payload_len;       /* ペイロード長 (NBO) */
    uint8_t  payload[POTR_MAX_PAYLOAD];
} PotrPacket;
```

## サービス定義ファイルの記述例

```ini
[service.1001]
type     = unicast
src_addr = 192.168.1.20
dst_addr = 192.168.1.10
dst_port = 5001

[service.1002]
type     = unicast
src_addr = sender.local
dst_addr = receiver.local
dst_port = 5002
```

## 設計判断のまとめ

| 項目 | 判断内容 |
| ---- | -------- |
| 受信側の bind アドレス | `dst_addr` を解決したアドレス (INADDR_ANY ではない) |
| 送信側の bind アドレス | `src_addr` を解決したアドレス (送信元 IP を固定) |
| 受信側の送信元フィルタ | `src_addr` と一致しないパケットをアプリ層で破棄 |
| IPv6 | 非サポート (IPv4 専用) |
| DNS 解決タイミング | `potrOpenService()` 時に 1 回解決し、プロセス内キャッシュ |
| DNS 更新への追従 | 非対応 (静的ルート前提、再接続はプロセス再起動) |
| 複数 DNS 応答 | 仕様上未定義 / 実装上は先頭アドレスを採用 |
| 送信元ポート | OS 自動選定 |
| 受信者の返信元ポート | `dst_port` (bind 済みソケットから送信するため) |
| service_id 照合 | 受信パケットの service_id とサービス定義を照合し、不一致は破棄 |
| セッション識別子 | `session_id` (uint32_t 乱数) + `session_start_time` (struct timespec NBO) |
| セッション採用ポリシー | `start_time` → `session_id` の順で大きい方を採用 |
| クロック逆行 | 既知の制限として仕様外 (静的環境前提) |
| 1:1 の制約 | 同一 IP からの複数送信者は仕様外 (将来対応予定) |
